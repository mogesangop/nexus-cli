# Nexus 主从 HA 模式用户故事拆分

> 来源：[nexus主从HA模式PRD.md](./nexus主从HA模式PRD.md)  
> 拆分方法：3C（Card、Conversation、Confirmation）+ INVEST  
> 范围约束：Nexus OSS 3.76、温备、人工切换、F5 不由 CLI 自动操作、RPO 非 0。

## 1. 角色

| 角色 | 核心任务 |
|---|---|
| DevOps 平台管理员 | 部署双节点、配置 F5、执行切换与回切 |
| 制品库管理员 | 配置和调优复制、验证 RPO 与权限一致性 |
| 运维/SRE | 巡检、故障判断、fencing、演练和审计 |
| nexus-cli 维护者 | 实现 HA 配置、命令、测试及向后兼容 |

## 2. Epic 与发布范围

| Epic | 目标 | 版本 | 优先级 |
|---|---|---|---|
| E1 HA 基础环境 | 建立满足单写约束的双节点温备架构 | v1 | P0 |
| E2 数据复制 | 将 blob 和元数据复制到从节点并可观测 | v1 | P0 |
| E3 人工切换与恢复 | 用 runbook 安全完成切换、验证和回切 | v1 | P0 |
| E4 HA 配置与状态检查 | 让 CLI 识别双节点并统一报告健康与延迟 | v2 | P0 |
| E5 同步与切换引导 | 用 CLI 执行末次同步并强制 fencing | v2 | P0 |
| E6 审计、兼容与质量 | 保持安全不变式和现有单节点行为 | v2 | P0 |
| E7 远期增强 | 降低 blob RPO、评估 F5 自动化 | v3 | P2 |

## 3. v1：运维落地故事

### HA-001 部署一致的主从节点

**Card**：作为 DevOps 平台管理员，我希望在两台独立 VM 上部署一致的 Nexus 节点，以便任一 VM 故障时仍有可接管节点。

**Conversation**：

- 两节点均使用 Nexus 3.76，分别承担 PRIMARY 和 STANDBY 角色。
- blob store 名称、目录结构及仓库配置必须一致。
- 从节点 Nexus 常驻运行，但不承接 F5 业务流量。
- 设计参考：原 PRD §7.1、§7.5。

**Confirmation**：

1. 两节点运行在不同 VM，且 Nexus 版本均为 3.76。
2. 两节点使用相同的 blob store 名称和目录结构。
3. F5 active pool 初始只有主节点，从节点不接收业务流量。
4. 运维可分别启动、停止和检查两个 Nexus 进程。
5. 客户端发布路径统一经过 F5，不存在已知的绕过 F5 写入路径。

**依赖**：无。

### HA-002 配置 F5 单节点流量入口

**Card**：作为 DevOps 平台管理员，我希望 F5 任一时刻只将业务流量发送给一个节点，以便落实单写原则。

**Conversation**：

- 本期仅提供 F5 配置模板和人工操作步骤。
- CLI 不调用 F5 iControl API，也不保存 F5 凭证。
- 设计参考：原 PRD §7.1、§7.3。

**Confirmation**：

1. F5 健康检查可独立显示主、从节点状态。
2. active pool 任一时刻最多包含一个可接收流量的 Nexus 节点。
3. 操作手册明确增加、移除及验证 pool member 的步骤。
4. 手工切换后，可从统一入口确认流量已到达目标节点。
5. 模板不包含真实密码、Token 或其他敏感凭证。

**依赖**：HA-001。

### HA-003 每 5 分钟复制 blob

**Card**：作为制品库管理员，我希望定时将主节点 blob 增量复制到从节点，以便将二进制制品 RPO 控制在 5 分钟内。

**Conversation**：

- 使用 rsync；仅当未来采用对象存储时再考虑 rclone。
- 仓库采用 `writePolicy: allow_once`，同步方向固定为主到从。
- 删除应同步到从节点，任务应可安全重跑。
- 设计参考：原 PRD §7.2.1。

**Confirmation**：

1. cron 每 5 分钟从主节点向从节点执行一次 blob 同步。
2. 同步命令覆盖新增、更新及删除，并能幂等重跑。
3. 同步失败返回非零状态并留下可定位的时间和错误信息。
4. 同步不从从节点反向覆盖主节点。
5. 测试制品发布后 5 分钟内可在从节点对应 blob 路径找到。
6. 同步通道使用环境已有的 SSH key 或内网认证，不在脚本中保存明文密码。

**依赖**：HA-001。

### HA-004 每 15 分钟复制元数据

**Card**：作为制品库管理员，我希望定时导出并导入 Nexus 元数据，以便将仓库、权限和组件索引的 RPO 控制在 15 分钟内。

**Conversation**：

- 主节点执行 Export database，从节点接收导出包并执行 Import database。
- 必须监控导出、传输和导入三个阶段，避免只看到“导出成功”而误判复制完成。
- 设计参考：原 PRD §7.2.2。

**Confirmation**：

1. 主节点每 15 分钟产生一次可用的数据库导出包。
2. 导出包完成后才传输，未完成文件不会被从节点导入。
3. 从节点成功导入后记录导入完成时间和导出批次标识。
4. 任一阶段失败均可被运维发现，且不会覆盖最近一次已知成功状态。
5. 仓库定义、组件索引、`priv_guest_*` 权限及 guest 角色可在从节点验证。
6. 连续执行两个周期不会产生并发任务堆积或不确定的导入顺序。

**依赖**：HA-001。

### HA-005 建立复制巡检与告警

**Card**：作为运维/SRE，我希望能看到两类复制任务的最后成功时间和延迟，以便在故障发生前发现从节点不可接管。

**Conversation**：

- v1 可使用任务日志、监控平台或巡检脚本；v2 再统一到 `ha status`。
- 告警必须区分 blob 与元数据复制。
- 设计参考：原 PRD §4、§10。

**Confirmation**：

1. 巡检结果分别展示 blob 和元数据的最后成功时间。
2. blob 超过 5 分钟、元数据超过 15 分钟未成功时触发告警。
3. Export 成功但 Import 失败时，元数据状态仍显示失败或滞后。
4. 从节点 Nexus 健康异常时触发独立告警。
5. 告警信息包含节点、任务、发生时间和可执行的排查入口。

**依赖**：HA-003、HA-004。

### HA-006 执行安全的故障切换

**Card**：作为运维/SRE，我希望按照检查清单切换到从节点，以便在 15 分钟内恢复服务且不产生双写。

**Conversation**：

- 顺序固定为：确认故障 → fence 旧主 → 尝试追赶同步 → 切 F5 → 服务验证 → 审计。
- 主节点硬故障时允许跳过末次同步并接受 RPO 内缺口。
- 设计参考：原 PRD §7.3。

**Confirmation**：

1. 检查清单要求先判断宕机还是网络分区。
2. 未确认旧主已停止或被隔离时，流程禁止进入 F5 切换步骤。
3. 旧主仍可访问时，切换前执行一次 blob 和元数据追赶同步。
4. 旧主不可访问时，检查清单明确记录跳过同步及可能的数据缺口。
5. 切换后匿名用户可通过 F5 下载指定验证制品。
6. 从故障确认到恢复匿名下载的总耗时可被记录并验证不超过 15 分钟。

**依赖**：HA-002、HA-003、HA-004、HA-005。

### HA-007 验证接管后的数据与权限

**Card**：作为制品库管理员，我希望在切换后验证制品、仓库和访客权限，以便确认从节点提供的服务符合预期。

**Conversation**：

- 验证包含正常下载、RPO 内缺口说明和 `guest check`。
- 个别制品 404 时，需要判断是同步窗口缺口还是配置错误。
- 设计参考：原 PRD §6、§9。

**Confirmation**：

1. 可匿名下载 `protected-repo-example` 中已完成复制的验证制品。
2. `nexus-cli guest check` 不报告 `priv_guest_*` 和 guest 角色漂移。
3. 仓库定义、online 状态及 blob store 引用与主节点最近快照一致。
4. 对缺失制品可关联到最后成功的 blob 和元数据同步时间。
5. 验证结果被附加到本次切换记录中。

**依赖**：HA-006。

### HA-008 安全回切并重建温备关系

**Card**：作为运维/SRE，我希望修复旧主后从当前主节点反向同步并回切，以便恢复稳定的主从保护能力。

**Conversation**：

- 数据方向以“当前承载写入的节点”为源，不能机械地按 A→B 执行。
- 目标节点在反向同步前必须停止或不可写。
- 设计参考：原 PRD §7.3 第 7 步。

**Confirmation**：

1. 回切前明确记录当前主节点和目标节点，禁止使用含糊的“原主/原从”指令。
2. 目标节点在反向同步前已停止服务或被可靠隔离。
3. blob 和元数据从当前主节点同步到目标节点并通过一致性验证。
4. F5 切换仍遵循先 fencing、后切流量的顺序。
5. 回切后重新建立单向定时复制并验证下一周期成功。
6. 回切全过程留下时间、执行人、方向和验证结果。

**依赖**：HA-006、HA-007。

### HA-009 完成故障演练与 RPO/RTO 验收

**Card**：作为 DevOps 平台管理员，我希望定期演练主节点故障，以便证明方案在真实环境中满足 RPO、RTO 和防脑裂目标。

**Conversation**：

- 首次发布前至少完成桌面推演、dev 实测、RPO 验证、权限验证和脑裂负向验证。
- 设计参考：原 PRD §9。

**Confirmation**：

1. 桌面推演记录每一步的执行人、输入、输出和升级路径。
2. dev 环境 kill 主节点后，匿名下载在 15 分钟内恢复。
3. 最近 15 分钟内发布的制品缺口被准确识别且不超过约定 RPO。
4. 从节点接管后的权限一致性通过 `guest check` 验证。
5. 未完成 fencing 时，演练流程确实阻止 F5 切换。
6. 演练结论、实际 RPO/RTO 和改进项形成可追踪记录。

**依赖**：HA-001 至 HA-008。

## 4. v2：nexus-cli 实现故事

### HA-101 扩展 HA 配置模型

**Card**：作为 nexus-cli 维护者，我希望在配置中声明 HA 节点、角色和复制策略，以便所有 HA 命令使用同一份受校验的配置。

**Conversation**：

- 新增 `ha.enabled`、`role`、`nodes`、`replication` 和 `failover`。
- 密码字段只保存环境变量名。
- 设计参考：原 PRD §7.4。

**Confirmation**：

1. 配置可声明且只能声明一个 primary 和一个 standby 节点。
2. `role` 只接受 `primary` 或 `standby`，`failover.mode` 本期只接受 `manual`。
3. 两节点名称、URL、`passwordEnv` 及复制方法和周期均被校验。
4. `requireFencing` 默认开启；关闭时给出明确的安全警告或拒绝生产切换。
5. 配置错误包含准确字段路径，不输出密码环境变量的值。
6. 示例配置可被 `config.Load` 和 `Validate` 成功处理。

**依赖**：无。

### HA-102 保持单节点配置向后兼容

**Card**：作为现有 nexus-cli 用户，我希望不配置 HA 时所有命令行为不变，以便升级 CLI 不影响当前治理流程。

**Conversation**：

- `ha` 缺省或 `ha.enabled=false` 时继续使用现有 `nexus.baseUrl`。
- 不改变 `guest sync`、`guest check`、`repo` 和现有 `health` 的语义。
- 设计参考：原 PRD §7.4。

**Confirmation**：

1. 现有配置文件无需增加 `ha` 段即可加载。
2. 现有命令使用相同 URL、凭证解析和退出码。
3. `guest sync` 的幂等行为和权限优先级不变。
4. `repo` 与单节点 `health` 命令的输出和网络请求目标不变。
5. 自动化测试覆盖 `ha` 缺省和显式关闭两种情况。

**依赖**：HA-101。

### HA-103 安全创建双节点客户端

**Card**：作为 nexus-cli 维护者，我希望为两个节点独立解析凭证并创建客户端，以便 HA 命令能同时访问主从节点且不泄露秘密。

**Conversation**：

- 延续“密码只从环境变量读取”的安全不变式。
- 节点错误必须标明节点名，但不能打印 Authorization header。

**Confirmation**：

1. 每个节点从其 `passwordEnv` 指定的环境变量独立读取密码。
2. 缺失凭证时错误包含节点名和环境变量名，但不包含密码值。
3. 两节点可使用不同的 base URL 和密码。
4. 日志、命令输出和错误中不出现密码或 Authorization header。
5. 客户端创建逻辑具有成功、缺失环境变量和非法 URL 的单元测试。

**依赖**：HA-101。

### HA-104 查看双节点健康状态

**Card**：作为运维/SRE，我希望运行 `nexus-cli ha health` 检查两个节点，以便快速判断主从节点是否可用。

**Conversation**：

- 复用现有 `internal/cli/health.go` 检查逻辑。
- 一个节点失败时仍应报告另一个节点结果。

**Confirmation**：

1. 命令分别报告 primary 和 standby 的节点名、角色及健康状态。
2. 对两个节点使用一致的健康检查规则。
3. 单节点超时或失败不会阻止另一个节点完成检查。
4. 任一节点不健康时命令返回非零退出码。
5. 输出不包含凭证或 Authorization header。

**依赖**：HA-103。

### HA-105 查看 HA 与复制状态

**Card**：作为运维/SRE，我希望运行 `nexus-cli ha status` 查看两节点健康和复制延迟，以便用一个入口判断从节点是否可接管。

**Conversation**：

- 状态至少包含节点健康、blob/元数据最后成功时间和复制延迟。
- “未知”与“成功”必须明确区分，不能把缺少状态误报为健康。
- 设计参考：原 PRD KR3、§10。

**Confirmation**：

1. 输出同时展示两个节点的名称、角色、URL 和健康状态。
2. 分别展示 blob 与元数据最后成功时间及当前延迟。
3. blob 延迟超过 5 分钟或元数据延迟超过 15 分钟时明确标红/告警并返回非零退出码。
4. 状态数据缺失、格式无效或读取失败时显示 `UNKNOWN`，不显示为成功。
5. Export 成功但 Import 未成功时，以 Import 完成时间计算元数据延迟。
6. 人类可读输出和自动化可消费输出使用同一状态模型。

**依赖**：HA-103、HA-104；需先确定复制状态的可靠数据源。

### HA-106 执行一次完整追赶同步

**Card**：作为运维/SRE，我希望运行 `nexus-cli ha sync --once` 触发一次 blob 与元数据同步，以便切换前尽量缩小一致性窗口。

**Conversation**：

- 同步方向必须从当前 primary 到 standby。
- 完整成功的定义是 blob 同步和元数据 Import 均成功。
- 该命令不得在方向不明或角色冲突时继续。

**Confirmation**：

1. 命令开始前验证配置中只有一个 primary 和一个 standby。
2. 按确定的顺序完成 blob 同步、元数据 Export、传输和 Import。
3. 任一阶段失败时立即报告阶段、节点和错误，并返回非零退出码。
4. 仅当所有阶段成功时更新对应的最后成功时间。
5. 重复执行不会破坏目标数据，且同步方向始终为 primary 到 standby。
6. 输出包含各阶段耗时和最终同步完成时间，不包含敏感信息。

**依赖**：HA-101、HA-103；需先确定 CLI 触发系统任务/外部 rsync 的执行接口。

### HA-107 强制执行 fencing 门禁

**Card**：作为运维/SRE，我希望 `ha failover` 在切流量指引前验证旧主已停止，以便防止网络分区或误操作造成双写。

**Conversation**：

- CLI 只做检查和引导，不直接操作 F5。
- “REST 不可达”不能单独证明已完成 fencing；需要可审计的人工确认或可靠的隔离证据。
- 设计参考：原 PRD KR4、§7.3。

**Confirmation**：

1. 命令先展示当前 primary、目标 standby 和预期同步方向。
2. 未获得 fencing 确认或可靠隔离证据时，命令拒绝进入后续步骤。
3. 仅检测到旧主 REST 不可达时，命令提示网络分区风险且不自动视为 fencing 成功。
4. fencing 通过后才允许尝试末次同步；旧主硬故障时可明确确认跳过。
5. CLI 只打印经过审核的 F5 操作指引，不调用 F5 API。
6. 测试覆盖旧主健康、旧主不可达、已确认隔离和用户取消四类路径。

**依赖**：HA-104、HA-106；需在实现前明确 fencing 证据/确认机制。

### HA-108 完成引导式人工切换

**Card**：作为运维/SRE，我希望 `ha failover` 按固定顺序引导我完成切换和验证，以便减少应急操作遗漏。

**Conversation**：

- 流程为：状态检查 → fencing → 末次同步或记录跳过 → F5 指引 → 接管验证 → 审计。
- Nexus 不执行 promote。

**Confirmation**：

1. 命令在开始时检查 standby 健康和复制延迟并显示风险。
2. 必须通过 HA-107 的 fencing 门禁才能显示 F5 切换步骤。
3. 末次同步失败时默认阻止继续；显式接受 RPO 风险后才可进入下一步。
4. 输出明确说明 Nexus 无 promote 操作，从节点已常驻运行。
5. F5 切换后对目标节点执行健康检查，并提示执行 `guest check` 和制品下载验证。
6. 任一步取消或失败均返回确定的退出码并记录实际完成到哪一步。

**依赖**：HA-105、HA-106、HA-107。

### HA-109 记录 HA 操作审计

**Card**：作为安全审计人员，我希望同步和切换操作写入统一审计记录，以便追溯谁在何时对哪些节点执行了什么操作。

**Conversation**：

- 复用 `audit.Record`，不增加密码或 Authorization header 字段。
- 需要记录结果，而不是只记录命令启动。

**Confirmation**：

## 7. 开发前必须澄清的产品决策

以下问题不影响 v1 runbook，但会直接影响 v2 的可实现性和故事估算：

1. **复制状态来源**：`ha status` 从 Nexus 任务 API、任务日志、状态文件还是监控平台读取最后同步时间？
2. **同步执行边界**：`ha sync --once` 是由 CLI 直接执行 rsync/SSH，还是触发外部编排任务？CLI 运行机器是否具备 blob 路径访问权？
3. **Import 能力验证**：目标 Nexus 3.76 的实际 REST/任务接口是否支持可靠触发和查询 Export/Import，需以目标实例 Swagger 和 dev 实测为准。
4. **fencing 证据**：采用人工输入确认、VM/容器状态检查，还是调用基础设施平台 API？仅 REST 不可达不能视为已隔离。
5. **非交互模式**：CI/自动化环境是否需要执行 `ha failover`；若需要，如何提供可验证而非简单 `--yes` 的 fencing 证明？
6. **状态输出格式**：是否需要 JSON，以供监控采集和自动化告警使用？

在上述决策明确前，HA-105、HA-106、HA-107 不建议直接进入开发承诺。

## 8. 全局完成定义（Definition of Done）

每个故事除自身验收标准外，还需满足：

1. 文档、示例配置和命令帮助与实际行为一致。
2. 所有失败路径有确定输出和退出码。
3. 日志、审计、错误和测试夹具均不包含密码或 Authorization header。
4. 新增 Go 逻辑有离线单元测试，且 `make test`、`make vet` 通过。
5. 不破坏现有策略优先级、托管权限边界和幂等行为。
6. 涉及运维流程的故事在 dev 环境完成一次可复现演练并保存证据。
